# バックエンド実装ルール 05 — Domain 適用優先度

## 基本方針

全機能を同じ深さでオニオンアーキテクチャにしない。

> 書込み系・状態遷移系・整合性制約が強い機能は初版から domain を強く適用する。
> 参照系・一覧系・集計系は軽量に始める。
> ただし依存方向だけは最初から守る。

---

## 優先度区分

### P0 — domain を最も強く適用する（最優先）

| 機能 | 理由 |
|------|------|
| 案件（job） | 他機能の土台。ステータス遷移制約と不変条件を持つ |
| 応募（bid） | 業務ルールの集中点。上限・締切・競合が複雑 |
| セット応募（bid 配下） | 最も破綻しやすい。オール・オア・ナッシング |
| 合意 / 成約（deal） | 不可逆操作。ドメイン中核 |
| 運送ステータス遷移（delivery） | 成約後の主要ステータス遷移 |

### P1 — domain を一部適用する

| 機能 | 理由 |
|------|------|
| 認証（auth） | ロックとトークン有効性は守る |
| テナント / ユーザー管理（tenant） | テナント境界は全機能の前提 |
| 連絡 / 最終条件提示（deal 配下） | 最終条件提示は成約に直結 |
| 評価（delivery 配下） | 1 回制約と状態変化がある |
| 通知（notification） | 発火条件は domain で守る |

### P2 — application + infrastructure 中心でよい

| 機能 | 理由 |
|------|------|
| ダッシュボード（query） | 参照・集計が中心 |
| 一覧検索（query） | read model 主体 |
| 取引履歴（query） | 取得・整形が中心 |

---

## 機能別 Domain 化の対象

### 案件（P0）

domain 化する:
- `JobStatus` とステータス遷移ルール
- 編集可能条件・削除可能条件
- 論理削除状態
- 成約後の更新禁止（不変条件）

domain 化しない: 高度な検索 DSL・地理情報の高度抽象化

### 応募（P0）

domain 化する:
- 1 社 1 提案制約
- 応募上限 20 社（`BidLimitPolicy`）
- 締切判定（from 時刻 2 時間前）
- 応募可能状態の判定
- 初回応募時の案件ステータス変化

domain 化しない: 応募検索の read model 最適化

### セット応募（P0）

domain 化する:
- 同一テナント制約
- セット構成の妥当性検証
- 全件成立 / 1 件不成立のオール・オア・ナッシングルール
- 他社成約時の連動クローズ判定

domain 化しない: セット応募履歴の分析用 read model

### 合意 / 成約（P0）

domain 化する:
- 合意可能状態の判定
- 成約成立条件
- スナップショット生成（`DealSnapshot`）
- 他応募クローズ条件
- セット応募との連動ルール

domain 化しない: 成約履歴の分析用 read model

### 運送ステータス遷移（P0）

domain 化する:
- 実行主体の制約（運送開始は運送会社のみ、完了確認は依頼企業のみ）
- 成約済 → 運送中 → 完了への遷移制約
- 完了確認の順序制約

domain 化しない: ステータス履歴表示専用の最適化

### 認証（P1）

domain 化する: ログイン失敗回数・ロック状態・トークン有効性

domain 化しない: メール送信文面の詳細制御・認証監査ログの高度抽象化

### テナント / ユーザー管理（P1）

domain 化する: テナント境界・ユーザー所属制約・同一企業重複登録判定の入口

domain 化しない: 高度な権限モデル・管理者 / 一般ユーザーの分離（V1 は権限分離なし）

### 通知（P1）

domain 化する: 通知発火イベントの定義・通知対象ユーザー/テナント範囲・通知種別の区分

domain 化しない: 高度な通知テンプレート管理・非同期キュー前提の配信設計

### 評価（P1）

domain 化する: 完了後のみ評価可能・1 回のみ登録可能・双方評価完了時の状態変化

domain 化しない: 評価分析集計・平均評価の公開設計

### ダッシュボード / 一覧 / 取引履歴（P2）

最低限 domain 化する: 参照可否のテナント境界・競合情報非表示などの表示制約

domain 化しない: CQRS の完全分離・分析向け集計専用ストア

---

## 実装順序

1. 案件・応募・セット応募・合意（P0）を最優先で domain 化する
2. 運送ステータス遷移・評価・通知（P0/P1）を続けて整える
3. 認証・テナント管理（P1）を並行して進める
4. ダッシュボード・一覧取得・取引履歴（P2）は軽量な query 実装で開始する
5. 運用開始後、変更頻度が高い箇所から domain を深める

---

## P0 機能の必須チェック

P0 機能を実装・レビューする際は以下を必ず確認する。

- [ ] 状態遷移ルールが Entity / Value Object に実装されているか
- [ ] 不変条件が Domain のメソッド内でガードされているか
- [ ] UseCase から直接 Infrastructure を呼んでドメインルールを飛ばしていないか
- [ ] 同時アクセス競合の考慮が UseCase またはトランザクション設計に含まれているか
- [ ] 通知発火が Domain Event / Application Event 経由になっているか
